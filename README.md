## Тестовое задание

Входящий звонок создаёт запись в таблице [calls], после чего в очередь Redis отправляется ProcessIncomingCallJob.
Job должен:
* найти клиента по номеру телефона;
* выбрать доступного оператора;
* назначить звонок оператору;
* отправить событие в телефонию;
* записать лог;
* при ошибке повториться.

Система работает в production под нагрузкой. Обработка звонков выполняется несколькими воркерами параллельно.

Фрагмент кода
```php
class ProcessIncomingCallJob implements ShouldQueue
{
    public $tries = 5;

    private $callId;

    public function __construct($callId)
    {
        $this->callId = $callId;
    }

    public function handle()
    {
        $call = Call::find($this->callId);

        if (!$call) {
            return;
        }

        if ($call->status === 'new') {
            $client = Client::where('phone', $call->phone)->first();

            if ($client) {
                $call->client_id = $client->id;
            }

            $operator = Operator::where('available', true)
                ->orderBy('last_call_at')
                ->first();

            if (!$operator) {
                throw new \Exception('No available operators');
            }

            $operator->available = false;
            $operator->save();

            $call->operator_id = $operator->id;
            $call->status = 'assigned';
            $call->save();

            // HTTP-запрос во внешнюю телефонию для назначения звонка оператору.
            // Гарантии внешней системы неизвестны.
            app(TelephonyClient::class)->sendCallAssigned($call->id, $operator->id);

            Log::info('Call assigned', [
                'call_id' => $call->id,
                'operator_id' => $operator->id,
            ]);
        }
    }
}
```

## Вопрос про масштабирование

Представьте, что через полгода нагрузка выросла в 10–50 раз: больше входящих звонков, больше операторов, больше параллельных воркеров, больше событий во внешнюю телефонию.

Опишите план масштабирования решения.

Нужно раскрыть:
* какие bottleneck-и вы ожидаете в текущей реализации;

* какие лимиты могут возникнуть в Redis, БД, HTTP-интеграции с телефонией и логировании; 


# Тестовое задание — разбор

## 0. Общие предположения

Я исхожу из того, что:

* Redis queue работает по at-least-once семантике, т.е. job может выполниться повторно
* воркеры запускаются параллельно и могут конкурировать за одни и те же данные
* внешняя телефония:

  * может тормозить или падать
  * не гарантирует идемпотентность
* оператор не должен получать больше одного активного звонка
* допустимы ретраи job, но недопустимы дубли назначения

---

# 1. Что здесь сразу вызывает риск (критичное)

## 1.1. Гонка за операторов

Сейчас выбор оператора:

```php
Operator::where('available', true)
    ->orderBy('last_call_at')
    ->first();
```

Проблема очевидная: при параллельных воркерах два процесса могут взять одного и того же оператора.

Я почти уверен, что это проявится под нагрузкой первым.

### Как бы я это начал фиксить

Самый простой шаг — заблокировать выбор:

```php
DB::transaction(function () {
    $operator = Operator::where('available', true)
        ->orderBy('last_call_at')
        ->lockForUpdate()
        ->first();
});
```

Но честно: я не уверен, что это выдержит высокую нагрузку.
Если начнутся lock contention — тогда уже нужно думать про переработку механизма выдачи операторов (например, через очередь/пул).

---

## 1.2. Нет атомарности между оператором и звонком

Сейчас:

* оператор помечается unavailable
* звонок сохраняется отдельно

Если что-то упадёт между ними — система остаётся в “полусломанном” состоянии.

### Как бы я сделал сначала

Просто обернул бы всё в transaction. Без усложнений.

Но здесь есть сомнение:
если внешняя телефония падает после этого — транзакция уже не спасает.

---

## 1.3. Нет идемпотентности job

Job может выполниться несколько раз:

* retry после exception
* повторная доставка из queue
* падение воркера после частичного выполнения

И самое опасное место:

```php
sendCallAssigned(...)
```

Может быть вызван несколько раз.

### Что бы я сделал

Я бы добавил явный “ключ назначения” (assignment id / uuid), и проверку на стороне call:

* если уже assigned → ничего не делаем
* если processing → тоже блокируем повтор

Честно: без поддержки на уровне данных это не решить полностью.

---

## 1.4. Нет защиты от двойной обработки звонка

```php
if ($call->status === 'new')
```

Это логически проверка, но не защита.

Два воркера могут одновременно пройти это условие.

### Более надёжный вариант

Я бы перевёл это в атомарный переход статуса:

```sql
UPDATE calls
SET status = 'processing'
WHERE id = ? AND status = 'new'
```

И уже по affected rows решал бы — кто “выиграл”.

---

## 1.5. Самое неприятное — потеря оператора при падении внешнего API

Сценарий:

* оператор помечен unavailable
* звонок записан
* вызов телефонии упал

И мы остаёмся в состоянии:

> оператор занят, но звонок не ушёл

### Честно

Я не стал бы усложнять это сразу компенсациями.

Но я бы заложил:

* логирование таких кейсов
* метрику “orphan assignments”
* и алерт

А уже потом решал бы:

* либо retry с восстановлением оператора
* либо отдельный reconciliation job

# 2. Важные, но не критичные проблемы

## 2.1. Нет backoff на retry

Сейчас retry может создавать нагрузочные пики.

Я бы добавил backoff, чтобы не устроить retry-storm.

---

## 2.2. Нет таймаута/защиты внешнего HTTP

Я бы явно добавил:

* timeout
* retry limit
* возможно circuit breaker

Но не уверен, что circuit breaker нужен сразу — зависит от стабильности телефонии.

---

## 2.3. Логи без контекста

Сейчас лог есть, но:

* нет correlation id
* нет связки request → job → external call

На масштабе это начнёт мешать дебагу.

---

## 2.4. Поиск клиента по телефону

Мелочь, но:

* нужен индекс
* иначе при росте базы начнётся деградация

# 3. Что бы я проверил в первую очередь (тесты)

Если бы я начал с тестов — я бы не распылялся.

## 3.1. Самое важное — конкуренция

* два воркера не должны назначить одного оператора
* два воркера не должны обработать один call

Это №1 риск системы.

---

## 3.2. Retry поведение

* повтор job не создаёт дубликат назначения
* повторный вызов телефонии не ломает состояние

---

## 3.3. Падение внешней системы

* телефония падает → система не уходит в неконсистентное состояние

---

## 3.4. Нет операторов

* корректная ошибка / retry
* не ломаем очередь

---

# 4. Что бы я НЕ стал делать сейчас

Я бы сознательно не усложнял систему:

* никаких микросервисов
* никакого event sourcing

# 5. Масштабирование (10–50x нагрузка)

Вот здесь я бы разделил честно: что сломается первым, а что — позже.

---

## 5.1. Что станет bottleneck

### База данных

Самое слабое место:

* конкуренция за операторов
* lock contention
* частые UPDATE на одни строки

Я ожидаю, что это упрётся первым.

---

### Redis queue

При росте нагрузки:

* queue lag начнёт расти
* retry storms могут усилить нагрузку

Но Redis обычно переживает лучше, чем DB в этом кейсе.

---

### Телефония

Очень вероятно:

* rate limits
* нестабильные ответы
* частичные ошибки

И это будет триггерить повторные попытки.

---

### Логирование

Если логи синхронные — начнут тормозить воркеры.

---

## 5.2. Что даст просто увеличение workers

Сначала — линейный рост throughput.

Но довольно быстро упрёмся в:

* блокировки в БД
* конкуренцию за операторов
* рост retry нагрузок
* лимиты телефонии

И дальше увеличение воркеров начнёт ухудшать ситуацию, а не улучшать.

---

## 5.3. Как бы я думал про масштабирование

Я бы разделил это на этапы.

---

### Этап 1 — стабилизация корректности

Сначала:

* atomic state transitions
* locks / transactions
* idempotency
* retry policy

Без этого масштабирование бессмысленно.

---

### Этап 2 — изоляция внешней системы

Я бы отделил:

* назначение звонка
* отправку в телефонию

Чтобы медленный внешний API не блокировал core обработку.

---

### Этап 3 — контроль нагрузки

Если начнутся проблемы:

* rate limiting на телефонию
* backpressure на очередь
* возможно DLQ для проблемных кейсов

---

### Этап 4 — переработка выдачи операторов (если станет узким местом)

И вот здесь уже возможны варианты:

* Redis-based pool
* preallocated assignment
* отдельный dispatcher

Но я бы не начинал с этого — это дорогая оптимизация.
